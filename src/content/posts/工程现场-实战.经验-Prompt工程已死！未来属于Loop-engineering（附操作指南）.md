---# Prompt工程已死！未来属于 Loop Engineering（附操作指南）

大家应该都有过这种经历：想让 AI 去检索论文，敲了一大段 prompt，它跑完了，一看，不符合我们要求。再补一段，它又跑，再看，还是差点意思。改到第五轮的时候开始怀疑一件事——到底是它在干活，还是我们在干活。

这件事在表面上是 AI 并没有达到我们的预期，实际却揭露了一个问题：我们的 prompt 很难指导 AI 达到我们想要的目标。

Anthropic 工程师、Claude Code 创始人 Boris Cherny 最近的一次分享中说道："现在，我觉得又到了下一个层级：我不再提示 Claude 了，我有一堆循环在运行，它们才是在提示 Claude 并判断接下来该做什么。我的工作变成了写循环。我认为，这是接下来几个月，甚至今年剩余时间里我们会看到的下一次转变。"

6 月 8 号，OpenClaw 的创始人 Peter 发了一条推：你不再需要为 coding agent 写 prompt 了。你应该设计循环来驱动你的 Agent。这个循环，也是 loop。

Boris Cherny 和 Peter 的公开讨论，正在把一个新范式推到台前：**Loop Engineering**——即开发者不再只是手动向编程 Agent 输入提示词，而是设计能够持续提示、调度和约束 Agent 的循环系统。

## Loop Engineering 到底是什么

谷歌云AI目前的总监，Addy Osmani 在6月9日发了一篇长文，把 Loop Engineering 这个概念正式梳理了一遍。

Addy Osmani 在这篇长文里先把一个完整的 loop 解释了一遍：这个loop是一套系统——我们定义好目标，构建一个能让AI自己找任务、执行、检查、记录、并且决定下一步，并且不断循环，直到完成的小系统。

Addy Osmani还在文中称：**Loop Engineering是未来我们与编程助手协作的新范式。**

接着，他把loop系统拆分为五个部分：

### 第一个：定时任务

这是整个loop的心跳，它的作用是自动启动循环，去完成我们既定的任务。它会让agent自动地去执行任务，如果途中发现问题，自动归为待处理队列；如果没有发现，自动归档。它与prompt最大的区别是：**我们不再是主动去检查的人。**

### 第二个：工作树隔离

它的作用是给每个 Agent 一个独立工作空间，防止文件冲突。当我们同时跑好几个 Agent时，应当给每个 Agent 各干各的，互不打扰，最后再进行汇总。因为如果让不同的Agent做同一个任务，他们会各做各的但不会互相沟通，做出来的结果就会互相覆盖，重复劳动。

### 第三个：Skills

它的作用是让Agent每次都能继承之前的知识体系。AI 每次开启新对话就什么都忘了。跟它说过的代码规范、项目架构、踩过的坑，每一次都要从零开始。所以得有一整套方法，也就是Skills来沉淀这些知识，让 Agent 每次启动后就了解这个项目。

这件事在 loop 里特别重要。因为 loop 是自动跑的，我们不在场。没有这些固定下来的规范、流程，就指导不了Agent按着项目进程工作。

### 第四个：MCP

它的作用是给Agent拓展外面的世界。一个只能看文件系统的 Agent，能力是非常有限的。因此我们需要给它接上 GitHub、Linear、Slack、数据库等等，它就能在真实工作环境里干活了。这才叫真正的闭环：从发现问题到解决问题到通知人类。

### 第五个：子 Agent

它的作用是将不同职责的Agent分开。写代码的 Agent 不能自己给自己打分，AI自己对自己都有包容性，总会从积极的一面看待自己的工作成果，这就很容易导致一个问题——里面有错误它发现不了。

这时候得用另一个 Agent，用不同的模型，专门检查前一个 Agent 的输出。当我们不在场时，这个"检查者"才是我们唯一能信任的守门人。我们常见的子Agent分工是：一个负责探索，一个负责实现，一个负责验证。

**这五个主要部分加在一起，就构成了一个 loop 的完整骨架。**

## 怎么用Loop Engineering检索论文

讲完了上面这些概念，大家是不是已经迫不及待地想要动手了，别着急。我再来给大家讲解下怎么用它来结合文献检索。

这个工程系统最适合的就是结合文献检索场景，因为：
- 第一，它有定时启动机制，能够帮助我们去每天定时地搜索对应内容的论文；
- 第二，它还告诉我们应当怎么去建立Skills管理，让Agent固定下来检索论文的流程、规范；
- 第三，MCP工具也可以帮助AI扩展外面世界，在真正的文献网站干活。

接下来我们将给大家分享，如何利用Loop Engineering结合到文献检索场景当中。

### 第一步：定时检索论文

整个文献循环的起点，由一个不知倦怠的后台任务唤醒。我们可以设定一个每隔 3 天在服务器后台自动运行的触发机制。它每次启动时，都会根据特定的一个主题和作者列表，定向去 arXiv、Google Scholar 以及各个顶级 AI 会议的最新接收列表中抓取当天新发布的论文。

这里我们以codex为例展示，只需要点击左上角的自动化。这里我们也给到一个以公司金融为主题的prompt，大家可以根据要求自己的主题进行修改。

```markdown
# Role
你是一个精通金融学与人工智能交叉领域的顶级资深审稿人。你的任务是审核后台自动化抓取到的最新论文，进行学术价值评估与定向筛选。

# Background
后台触发器每 3 天会推荐一批包含"公司金融 (Corporate Finance)"或相关作者的最新论文。由于部分检索结果存在噪音，你需要作为第一道智能防线，剔除无关文献，锁死真正具备学术价值的研究。

# Screening Criteria (筛选标准)
请仔细阅读论文的 Title 和 Abstract，判断其是否属于以下核心关注圈：

1. **重点关注**：
   - 传统公司金融核心：资本结构、公司治理、兼并收购（M&A）、企业投资决策、薪酬激励、现金流管理
   - 金融科技与 AI 赋能：利用大语言模型（LLM）、机器学习处理公司年报（10-K）、分析管理层讨论与分析（MD&A）、预测企业违约风险、绿色金融（ESG）的数据挖掘

2. **严格过滤**：
   - 纯粹的二级市场量化交易策略（如多因子模型、高频交易）
   - 与公司微观决策无关的宏观经济学、纯粹的计算机硬件/底层算法改进论文

# Execution Workflow

- **Step 1 [相关性评估]**：判断论文是否通过上述筛选标准。若不通过，直接输出 `[STATUS]: REJECT` 并简述原因（1句话）。
- **Step 2 [高价值提取]**：若通过，输出 `[STATUS]: KEEP`，并立刻调用工作区根目录的 `SKILL.md`，按照既定的金融学术锚点提取：
  - 核心研究问题（Research Question）
  - 关键理论假说或实证方法（Methodology/Data Source）
  - 核心贡献与创新点（Contributions）

# Output Format

[STATUS]: KEEP / REJECT
[Title]: 论文标题
[Review]: (若 KEEP，请在此输出结构化的关键信息提取)
```

### 第二步：论文解析工作隔离

当自动化任务把新论文下载到服务器后，Agent 需要立刻启动大规模的 PDF 解析、图表提取和文本向量化工作。

比如：一篇需要Python Marker环境，另一篇需要Java的Grobid环境。这时如果同时处理多篇论文，很容易产生依赖冲突。这时需要介入工作树隔离机制。系统自动为这两篇论文分别派生出独立的 Git Worktree。

**配置操作：** 在Agent的系统指令中，为其设定一套固定的隔离解析流程。告诉Agent在拿到新论文时，必须按以下步骤执行命令：

```bash
# Agent后台执行的逻辑示例

# 1. 为新论文A创建独立的工作区
git worktree add ../paper-A-parse main

# 2. 进入隔离区
cd ../paper-A-parse

# 3. 环境判断分支
if [ "包含大量公式" ]; then
    docker run -t --rm -p 8070:8070 lfoppiano/grobid
elif [ "扫描版/大量插图" ]; then
    # 启动Grobid服务解析结构
    marker_single paper-A.pdf --output_dir ./results
    # 启动Marker进行OCR
fi
```

**运行逻辑：** 用Git Worktree代替传统的开新文件夹，好处是所有的解析脚本（代码）在主仓库更新后，各个隔离区能实时同步，而解析产生的数据和临时文件互不干扰，结束后直接 `git worktree remove` 即可无痕销毁。

### 第三步：论文提取Skills

进入隔离的工作区后，Agent 会对清洗出来的论文文本进行深度泛读和关键信息提取。这时候有个问题，如果它没有任何知识储备，它会盲目地去做一些泛泛的摘要，或者错过了我们真正关心的学术问题。

为了防止它在这些方向性问题上反复试错，我们应当提前写好 `SKILL.md` 并且让它立刻读取并调用。这里我也把提前写好的论文提取prompt给到大家：

```markdown
# Role & Goal
你是一个顶尖的学术论文深度泛读与特征提取专家。你即将进入隔离工作区处理论文原始文本。

# Execution Rules

1. **拒绝盲目摘要**：严禁在没有任何指引的情况下进行宽泛、无重点的泛泛摘要。
2. **技能优先调用**：在对文本进行任何处理前，必须[立刻读取并严格遵循]工作区根目录下的 `SKILL.md` 文件。
3. **专家视角对齐**：将 `SKILL.md` 中定义的学术关注点、专业定义、特定指标及过滤规则，作为你阅读和提取的唯一最高准则。

# Workflow

- **Step 1**: 优先加载 `SKILL.md`，锁定本次阅读的核心"学术锚点"与"提取范式"。
- **Step 2**: 带着明确的学术问题深度扫描论文文本，精准提取符合 SKILL 要求的关键信息。
- **Step 3**: 严格过滤掉与 SKILL 无关的冗余信息，输出高含金量的结构化分析。
```

### 第四步：学术生态MCP工具

当文献的关键情报被精准提炼出来后，我们需要让Agent通过MCP长出"触手"，直接操作我们的工具链：

- 用MCP调用文献管理插件，自动将这三篇论文的完整元数据放进我们的Zotero文献库中
- 通过MCP连接到我们的项目管理工具或实验室主页，更新科研看板
- 调用通知插件，通过 Slack/飞书 告诉我们论文已经筛选完成

这里我们以配置Zotero和飞书的MCP连接器为例，假设已安装相应的开源MCP Server：

**配置操作：** 通过配置MCP Server的JSON文件，让Agent获得操作其他软件的权限。

```json
{
  "mcpServers": {
    "zotero-connector": {
      "command": "node",
      "args": ["/path/to/mcp-zotero/build/index.js"],
      "env": {
        "ZOTERO_API_KEY": "你的Zotero密钥"
      }
    },
    "feishu-notifier": {
      "command": "python3",
      "args": ["/path/to/mcp-feishu/main.py"],
      "env": {
        "WEBHOOK_URL": "你的飞书群机器人链接"
      }
    }
  }
}
```

### 第五步：核对Sub-agents

当报告生成之后，这还不能算完。为了防止主Agent"轻信"原作者的结论，我们需要在最后收尾的验证阶段，设置reviewer子智能体机制来进行对检索的文章进行核对和总结。

**配置操作：** 在流程的收尾脚本中，将提取好的报告交给一个被设定了极其严苛Prompt的 Reviewer Agent进行二次审核。

```bash
# 将主Agent的输出(report.md)管道传输给Reviewer Agent
cat report.md | claude -p "
你是一个顶刊（如 Springer/Nature 级别）的苛刻审稿人。
你的任务是审查这份文献总结：
1. 质疑其创新点：作者声称的性能提升是否掩盖了某些假设条件？
2. 查找逻辑漏洞：研究方法（如采用的质性编码或回归模型）与结论之间是否存在推导断层？
输出一份红色的【避坑指南】，追加到原报告的末尾。
" > final_report.md
```
