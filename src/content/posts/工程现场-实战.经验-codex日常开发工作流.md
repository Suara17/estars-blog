## 背景
来源：实习经验分享  
时间：2026-06-05

## 工作流概览

每天到办公室的标准化流程：

### 1. 环境准备
- 打开电脑
- 启动 **4 个 Codex 实例**：
  - 1 个 VSCode 插件
  - 1 个 Desktop 客户端
  - 2 个终端（终端 A、终端 B）

### 2. 需求对齐与规划（Desktop）
- 在 Desktop 中对齐需求
- 进行任务拆解和规划
- **充分讨论**后，让 Codex 生成一套 prompt

### 3. 执行阶段（终端 A）
- 使用终端 A 开 **plan 模式**执行
- Codex 的 plan 几乎直接执行，有时甚至不看、不拒绝

### 4. 代码审查（终端 B + VSCode 插件）
- **终端 B**：使用 `/review` 命令进行 review
- **VSCode 插件**：使用针对此次任务的定制 prompt 来 review，追踪代码

### 5. 交付
- 审查通过后直接 push，创建 PR

## 实际操作中的体会

### 关于 plan 模式

终端 A 开 plan 模式执行的时候，Codex 的 plan 几乎直接执行，有时甚至不看、不拒绝。这意味着：
- **规划质量决定了执行质量** — 在 Desktop 里讨论得越充分，plan 模式跑得越顺
- **有时候 plan 模式会自作主张** — 遇到模糊的需求它会自己脑补，所以执行完一定要 review

### 关于 review 的坑

终端 B 用 `/review` 命令，VSCode 插件用定制 prompt 来 review，这两个视角是互补的：
- 终端 B 的 `/review` 偏向语法和逻辑检查，比较机械但稳定
- VSCode 插件的定制 review 偏向架构和一致性检查，能发现 `/review` 发现不了的问题
- **推荐做法**：两个都跑一遍，先 `/review` 再过插件，交叉验证

### 关于"所有问题都问 Codex"

这个原则一开始我觉得有点极端，实际用下来发现是对的：
- 遇到编译报错 → 贴给 Codex，它秒解
- 遇到业务疑问 → 贴给 Codex，让它结合上下文分析
- 遇到不确定的技术选型 → 让 Codex 列优缺点，自己做决策

**但有个前提**：你要能判断 Codex 给的答案靠不靠谱。所以它不是替代你的判断力，而是放大你的判断力。

## 核心原则

1. **代码从不手写** — 所有代码都由 Codex 生成
2. **所有问题都问 Codex** — 碰到的任何问题，无一例外
3. **充分讨论后再执行** — 规划阶段花足够时间，执行阶段高效推进
4. **双重视角的 review** — 终端 `/review` + 插件定制 review，交叉验证
5. **人要保留判断力** — Codex 是放大器，不是替代品

## 工作流图示

```
Desktop (需求对齐/规划)
    ↓ 生成 prompt
终端 A (plan 模式执行)
    ↓ 代码产出
终端 B (/review) + VSCode 插件 (定制 review)
    ↓ 审查通过
git push → PR
```

## 关键思考

> 这样会有提升吗？

这套流程的本质是把 **人的角色从"写代码"转变为"审代码"**，核心价值在于：
- 规划阶段充分思考，减少返工
- 多维度 review（终端 + 插件交叉验证）
- 所有问题都问 Codex，持续积累 prompt 经验
