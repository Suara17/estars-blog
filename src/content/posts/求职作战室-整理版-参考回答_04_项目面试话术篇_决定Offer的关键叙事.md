## 💡 为什么这 7 道题最关键？

面试的大部分时间都在聊项目。面试官通过项目叙事判断你的**真实能力水平**——不是看你知不知概念，是看你是不是真的做过、理解多少、能多深。

这 7 道题几乎是所有公司面试的必考题，而且**每一道都是追问起点**。答好了，面试官会对你产生"这人靠谱"的印象；答崩了，后面你八股背得再顺也救不回来。

### 题1：请用"业务痛点→初版方案→踩坑迭代→量化收益"介绍项目

**参考回答（面试口语化——四步公式，决定性题目）：**

这道题是面试中的"第一印象题"，决定了面试官对你这整个人的判断基调。

推荐用 **四步公式** 来讲：

**第一步：业务痛点（说清楚"为什么要做"）**
> "在我们的开发流程中，代码审查和重构占了大量时间。尤其是一些跨模块的修改，老手改了都要反复检查，新手改完了经常出 bug。我们团队平均每次重构评审要 2-3 小时，而且漏审率并不低。所以我们就想——能不能用 AI Agent 来自动化这部分工作，帮开发者省时间、降低出错率。"

**第二步：初版方案（怎么做的，但不要只讲"用了什么"）**
> "第一版我们参考了 Claude Code 的架构思路，做了一个基本的 code agent——给它定义了代码搜索、文件读取、代码修改三个核心工具，用 ReAct 范式来驱动。LLM 用的是 GPT-4o，工具调用走 Function Calling。一开始效果还行，简单的代码审查能完成，但复杂场景下容易跑偏。"

**第三步：踩坑迭代（关键！展示你遇到过真实问题）**
> "跑了半个月发现几个问题：① Agent 在长流程中容易忘了自己改到哪了——需要做状态管理；② 工具返回结果的 token 占用太大，3-4 步就把上下文塞满了；③ 模型生成的代码有时能用但不符合公司的 coding style。
>
> 针对这些问题我们做了三轮迭代：
> - 第一轮：加了上下文压缩机制，把工具返回的日志信息用摘要替换，token 消耗降了 35%
> - 第二轮：加了公司编码规范的校验工具——Agent 修改完代码后自动跑一次规范检查，不过关就 mark 出来重新改
> - 第三轮：加了执行状态追踪，每步记录当前进度，出错了能从中断的地方恢复，不用从头开始"

**第四步：量化收益（必须有数据！）**
> "优化后效果明显：单次任务 token 消耗从 8,500 降到 5,200（压了 38%）；任务完成准确率从 66.7% 提升到 92%（benchmark 测试）；工具的响应速度平均快了 1.8 秒。更重要的是——我们的开发者每周平均省了 4 小时的代码审查时间，这个时间可以去做更核心的事情。"

> **高分关键**：踩坑迭代环节要有**真实感**——不要说"遇到了一些问题"，要说"遇到了 XX 具体问题，我们走了 XX 弯路，最终用 XX 方式解决"。

### 题2：你这个项目到底在解决什么问题？

**参考回答（面试口语化——区分"toy project"和"真实工程"的关键）：**

这个问题比题1更聚焦——它要求你用**一句话说清楚项目的核心价值**。

好的回答是这样的：
> "这个项目解决的是**开发效率与代码质量之间的矛盾**——在保证代码质量不下降的前提下，提升开发者的代码审查和重构效率。"

为什么这个回答好？因为它点出了**本质矛盾**，而不是描述功能：
- ❌ 错误示范："我做了个 code agent，它能自动化代码审查"
- ✅ 正确示范："我做的 code agent 解决的是'又快又好'的矛盾——传统流程中速度和准确率是 trade-off，Agent 可以同时兼顾"

**面试官想听到的三层：**
1. **这个场景有什么痛点？**（效率低 / 容易出错 / 成本高）
2. **痛点背后的本质矛盾？**（速度 vs 质量 / 通用 vs 定制 / 自动 vs 安全）
3. **你的方案怎么解了这个矛盾？**（用 Agent 的闭环能力替代人工低效环节）

> **加分技巧**：说出来后补一个"如果没有这个方案会怎样"的反例——"如果没有 code agent，一次跨模块的重构可能需要 2 个人花一整天做 review，而且覆盖率还不高。"

#### 🔗 补充：系统为什么不会失控？（校招生最容易接不住的追问）

**参考回答（面试口语化）：**

这是校招生做 Agent 项目时最容易被问住的问题——"你给了 Agent 执行代码、操作文件、调 API 的能力，它乱来怎么办？它失控了你怎么兜底？"

我的回答分三个层面来拆，每一层都能打消面试官的疑虑：

**第一层：入口限制——Agent 没有"想做就做"的自由度**

Agent 所有的能力都是**显式注册**的。你在 tool registry 里注册了什么，Agent 才能用什么。没有注册的工具，Agent 没法凭空调用。所以不存在"Agent 突然自己学会了调危险接口"这种事情。而且每个工具的 description 里会说清楚使用限制——比如"文件删除工具只允许删除工作区目录下的文件"，Agent 即使被注入攻击了，它的工具调用范围也是固定的。

**第二层：执行隔离——Agent 在"沙箱"里干活**

实际项目中所有工具有执行隔离：
- **文件系统隔离**：Agent 只能访问工作区目录，不能碰系统文件。用 Docker 挂载卷 + 用户映射实现——Agent 在容器里是 nobody 用户，没权限改系统配置。
- **网络隔离**：非必要的网络请求会被拦截（比如 Agent 不应该访问内网数据库就在网络层封掉）。
- **操作审计**：每一步工具调用都记录了完整的入参、出参、时间戳。出了事可以复盘——"Agent 在哪个环节做了什么事、为什么这么做"。

**第三层：人在回路——关键操作永远有人确认**

对于有破坏性的操作（删除文件、执行高危命令、写数据库），Agent 不能直接执行——它只能生成操作方案，等用户确认后才执行。这叫"门槛机制"：Agent 可以建议，但不能拍板。等于说 Agent 的自主权不是无限的，它其实是一个"聪明但需要上级签字的下属"。

另外还有一层——**运行时的异常熔断机制**前面讲过了（工具连续失败 3 次自动停、超时自动截断、循环调用检测），这些不是在"假设 Agent 正常工作"时的设计，而是在"假设 Agent 已经不正常了"时还能兜住。

> **面试加分总结**：面试官问"失控"其实是在问——"你是不考虑后果就做了个 Agent，还是认真想过安全边界？"你只要从"注册限制""执行隔离""人在回路"三个角度讲清楚，他就知道你是认真上过生产、不是随便写个 demo 了。再补一句："我们实际跑下来发现，90% 以上的'失控'场景其实不是 Agent 恶意了，是工具调用参数传错了——所以我们重点优化了参数校验层，让不合法参数在调用工具前就被拦截。"

### 题3：为什么这里要做成 Agent？不是直接用现成方案？

**参考回答（面试口语化——校招生最怕的追问，答案要有深度）：**

这个问题是面试官在判断：**你是为了用 Agent 而用 Agent，还是真的需要 Agent。**

**回答核心思路：区分"Agent 能做的"和"非 Agent 不可的"**

> "其实第一版我们想过用现成方案——比如 Claude Code、GitHub Copilot 直接对接。但跑通 POC 后发现了几个问题：
>
> **为什么 Claude Code 不够用？**
> Claude Code 是通用方案，它不了解我们项目的代码规范、依赖关系、业务流程。比如让它改一个模块的接口，它改了 A 文件不知道还要同步改 B、C 文件，生成的代码风格也不一致。
>
> **为什么 Workflow 不够用？**
> 因为代码审查和重构这件事**不确定性太高**——每次传进来的代码都不一样、改动的逻辑不同、依赖关系千变万化。Workflow 能处理固定流程（比如"从 API 抓数据→清洗→存库"），但代码审查需要 Agent 的**动态决策能力**：看到一段代码，判断它有没有问题，如果问题严重怎么改——这东西没法用固定流程写死。
>
> **那为什么是 Agent 不是人？**
> 因为重复性的审查工作占用开发者大量时间，而且人也有注意力疲劳的问题。Agent 可以 7x24 小时跑、保持一致的审查标准、还能积累审查经验（长期记忆跨会话复用）。"

**一句话总结：**
> "用 Agent 是因为它解决了'不确定性任务自动化'的问题——Workflow 做不了，人做太慢，只有 Agent 能在这之间取得平衡。"

> **加分关键**：不要说"用 Agent 更酷"——要用"为什么 Workflow 不够、为什么现成方案不行、为什么人来做不好"来说服面试官。

### 题4：自己做 Code Agent 的动机？和 Claude Code 比差异？

**参考回答（面试口语化——字节高频题）：**

> **动机**：现有 code agent（Claude Code、Cursor）是通用工具，它们的设计目标是"覆盖尽可能多的场景"，但这也意味着在特定业务场景下表现不够好。我做 code agent 的动机是通过**定制化**来提升特定场景的效果——就像你不能指望一个瑞士军刀能干好手术刀的活。
>
> **和 Claude Code 的差异**：
>
> 1. **能力边界不同**：Claude Code 是全能的——能分析代码、修改、调试、部署。我们的 agent 聚焦在"代码审查 + 自动化重构"两个场景，不做部署和调试。边界收窄了，但在这两个场景下的准确率和稳定性比通用方案高很多。
>
> 2. **上下文注入不同**：Claude Code 的上下文主要来自代码库本身。我们的 agent 还额外注入了公司的编码规范、项目依赖图谱、历史审查记录。这让它"更懂我们"。
>
> 3. **工具链差异**：我们的 agent 的工具是定制开发的——代码搜索用语义搜索（自己的 embedding + 向量库）、代码修改走公司内部的代码管理平台 API、审查结果直接关联到我们的 issue 系统。这些 Claude Code 做不了。
>
> 4. **成本控制**：通用方案 token 消耗大。我们通过上下文压缩、缓存、轻量模型组合，成本只有通用方案的 40%。

> **面试加分关键**：不是要踩 Claude Code 不好，而是要说"Claude Code 很好，但通用方案不能解决我的特定场景，所以我做了针对性优化"——体现判断力。

### 题5：你怎么看待 AI 编程？如何把握自己的能力和边界？

**参考回答（面试口语化——滴滴/多家中小厂常问）：**

> 我的视角分两个层面：
>
> **正面**：AI 编程极大地提升了开发效率——代码补全让我少敲 30% 的 boilerplate，Agent 能帮我处理重复性的修改和审查任务，让我更专注于有挑战性的设计工作。这不是"取代开发者"，而是"让开发者从重复劳动中解放出来"。
>
> **但我要说清楚自己的边界**：
>
> 1. **AI 生成的代码我会 review，从不直接 merge**——这跟人写的代码要 review 一样的，AI 可能逻辑正确但隐含着 bug，或者用了错的 API
> 2. **核心架构设计我做，AI 只是辅助建议**——比如系统架构、模块划分、数据流设计，这些依赖业务理解和工程经验，AI 给的建议我会参考但不会盲信
> 3. **我知道 AI 的幻觉边界**——跟 AI 确认"这个 API 有没有"不会直接复制文档，而是用它提供的线索去查官方文档验证
>
> **我的原则**：AI 是工具，不是替代品。用它提升效率，但保留自己的判断力。面试官问这个问题，其实是看候选人**有没有 AI 依赖症**。

> **加分回答**：加一个自己的实践——"我每周会总结一次这周用 AI 做了什么、哪些是 AI 做得好的、哪些是 AI 翻车的，这样就能逐渐摸清 AI 的能力边界和我的能力边界。"

### 题6：AI 工具出现后，对传统研发工作流造成了什么冲击？

**参考回答（面试口语化——考察深度思考）：**

> 冲击是全方位的，我概括成四个层面：
>
> **① 开发效率的提升**：以前一个需求从设计→编码→CR→测试→部署，每个环节都是串行的。现在 AI coding 工具让编码和部分测试可以大幅度提速——原来 2 天的编码工作可能半天就完成了。流程的瓶颈从"写代码太慢"变成了"方案设计不够充分"。
>
> **② 代码质量的定义在变化**：以前质量主要看代码的风格、性能和可维护性。现在 AI 生成的代码质量差异很大——有时它生成的质量比人高（因为它见识过的代码比人多），有时会有奇奇怪怪的 bug。所以 CR 的标准需要调整：不再审查"风格问题"，集中在"逻辑正确性"和"架构一致性"上。
>
> **③ 开发者的角色在进化**：程序员不再是"写代码的工具"，而是"选择和管理 AI 输出的人"。你会什么不再重要，你能不能让 AI 产出高质量代码更重要。这其实提高了对开发者"设计能力"和"判断力"的要求。
>
> **④ 安全问题变复杂了**：AI 生成的代码可能有安全隐患——依赖注入、敏感信息泄露、许可证冲突。安全审查需要跟上。
>
> **我的核心观点**：AI 工具不是在"取代"开发者，而是在"筛选"开发者——能驾驭 AI 的人越来越强，不能驾驭 AI 的人慢慢被边缘化。

> **加分回答关键**：不要只说宏观变化，加一个自己工作中的实际体会——"举个具体例子，我们团队……"

### 题7：你知道这个方向上已经有什么成熟方案吗？差在哪？

**参考回答（面试口语化——技术敏感度+判断力）：**

> 在 code agent 这个方向上，我调研过的主流方案有几个：
>
> **① Claude Code（Anthropic）**
> 目前最成熟的 coding agent。优点：多文件编辑、错误恢复、上下文管理都很强。不足：通用方案，对特定业务的代码规范/框架/工具链不了解；token 消耗大，成本偏高；没有审计追踪。
>
> **② Cursor（AI-native IDE）**
> 体验最好的 AI IDE。优点：深度集成、Tab 补全质量高、代码库理解强。不足：是 IDE 层面的方案，无法做成服务/集成到 CI/CD。本质上还是在帮开发者写代码，不是自动化执行代码任务。
>
> **③ Devin（Cognition）**
> 全自主开发的 Agent。优点：全流程覆盖（需求→编码→部署），工程集成好。不足：目前能力还不稳定，复杂任务失败率高；不支持定制化；需要联网部署。
>
> **④ LangChain 的 Code Agent**
> 开源框架，灵活性好。优点：可定制化程度高、社区活跃。不足：调试困难、生产环境稳定性不够、Agent 容易跑偏。
>
> **我们的差异**：这些方案各有侧重点，但在"公司内部代码规范 + 特定框架支持 + 审计追踪 + 低成本"这个组合上，没有一个现成方案能满足。所以我们做了针对性优化——不追求全场景覆盖，而是在"代码审查+重构"这个场景下做到最好。

> **加分关键**：不要只说"我知道这些方案"，要说"我知道这些方案的**定位和取舍**，以及我的方案为什么在这个取舍点上做了不同的选择"。

## 🎯 项目话术的终极心法

### 1. 四步公式背熟
```
业务痛点 → 初版方案 → 踩坑迭代 → 量化收益
```
每一步都要有**具体细节**，不要只喊概念。

### 2. 准备 2-3 个"踩坑故事"
面试官最想听的就是这个——要准备：
- 一个**技术选型**的坑（"我们选了方案 A 但后来发现……"）
- 一个**工程落地**的坑（"上线后发现 token 消耗远超预期……"）
- 一个**边界场景**的坑（"用户一个长文本直接把上下文塞爆了……"）

### 3. 任何一句话都要能被追问
不要讲"我们做了优化"这种空话——要讲"我们做了 XX 优化，通过 XX 方法，带来了 XX 效果，过程中遇到了 XX 问题"。面试官每个细节都可以往下追问。

### 4. 量化！量化！量化！
- ❌ "效果还不错"
- ❌ "感觉好了很多"
- ✅ "准确率从 66.7% 提升到 100%"
- ✅ "token 消耗降了 38%"

### 5. 准备"反过来"的回答
面试官可能追问："那为什么不选 B 方案？"——要准备好备选方案的对比分析，每个技术选型至少 2-3 个理由。
