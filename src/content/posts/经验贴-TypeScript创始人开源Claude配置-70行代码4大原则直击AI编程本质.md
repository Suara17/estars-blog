## 二、4 大原则深度解读 🧠

### PRINCIPLE 1: The Agent Didn't Do What I Want（特工没有按我所愿行事）

**这条最重要，也最容易被人忽视。**

Andrey 的原话是："没人确切知道自己想要什么，这在 AI 时代也是如此。"

你和 Agent 之间可能存在沟通障碍，所以需要进行盘问，让 Agent 详细询问你正在建造的东西。

- 😣 很多人抱怨："AI 给我写的东西不是我想要的！"
- 🤔 真相是：你根本没想清楚自己要什么
- ✅ Andrey 的解法：让 AI 反过来盘问你，逼你把需求说清楚

**核心洞察：** AI 写代码的失败，80% 不是 AI 的问题，是需求问题。

实战写法：
```
# Principle 1: The Agent Didn't Do What I Want
- 遇到任何模糊需求，必须先盘问我
- 不要假设我想要什么
- 主动追问：
  · 我要解决什么问题？
  · 输入输出是什么？
  · 边界条件是什么？
  · 有没有类似参考？
- 只有需求清晰后，才能开始写代码
```

---

### PRINCIPLE 2: Think Before You Code（先思考再编码）

这条是顶级程序员的"老黄历"，但 90% 的人做不到。

Andrey 的意思：
- 🛑 不要上来就敲代码
- 🧠 先思考：架构、边界、异常、设计
- 📝 写下来：把思路、方案、权衡写清楚
- ✅ 确认方案后再动手

Andrey 的原话："你看到一个需求，第一反应不应该是打开 IDE，而是拿起纸笔（或者打开文本框）把方案想清楚。"

写代码前先写 200 字的方案，比写完代码再改 200 行，效率高 10 倍。

实战写法：
```
# Principle 2: Think Before You Code
- 任何复杂任务开始前，必须输出方案：
  · 设计思路
  · 关键模块拆分
  · 边界条件 / 异常处理
  · 性能考量
- 方案得到确认后，才能开始写代码
- 简单任务可跳过，但默认走流程
```

---

### PRINCIPLE 3: Simplicity First · 精准修改

这条是最容易被忽视的"内功"。

很多 AI 写代码的毛病：
- 😤 过度抽象：一个简单功能写出 5 个 class
- 😤 过度设计：硬塞一堆模式、接口、泛型
- 😤 大刀阔斧：让你修一个 bug，它重写整个模块

Andrey 的解法：
- ✅ 能简单就不要复杂
- ✅ 精准修改：只动需要动的地方
- ✅ 不要改无关代码
- ✅ 新功能 ≠ 重构

Andrey 的警告："AI 最容易犯的错 —— 改一个 bug 顺手重构整个文件。你打开 PR 一看，diff 2000 行，90% 的改动是 AI 顺手干的，reviewer 直接崩溃。"

实战写法：
```
# Principle 3: Simplicity First · 精准修改
- 不要过度抽象
- 不要顺手重构
- 只改需要改的代码
- 禁止在修复 bug 时改无关逻辑
- 禁止在加新功能时顺手重构
- 修改前先看当前代码，不要凭想象改
```

---

### PRINCIPLE 4: Goal-Driven Execution（目标驱动执行）

这条是"反 AI 摸鱼"的核武器。

- 🎯 目标要明确、可量化（不是"做好"而是"做到 100% 覆盖"）
- 📋 列验收标准（写之前先说"什么算完成"）
- 🔁 自检 + 反馈循环（每步检查自己是否达标）
- ⛔ 未达标 = 不准停

核心思想："不是'做到哪算哪'，是'不达到 100% 验收标准不准停'。"

实战写法：
```
# Principle 4: Goal-Driven Execution
- 任何任务必须有明确目标 + 验收标准
- 没达成的不准收工
- 每一步都要自检，不合格的重做
- 遇到无法完成的部分：
  · 禁止跳过
  · 禁止写"待查"
  · 必须说明原因 + 给出替代方案
```

## 四、为什么这 4 条能火？底层逻辑 🔍

| 原则 | 解决的痛点 | 对应场景 |
|------|-----------|---------|
| 1. 需求盘问 | 需求模糊 → AI 瞎写 | 改 bug 改出新问题 |
| 2. 先思考再编码 | 直接动手 → 方案错误 | 推翻重做浪费时间 |
| 3. 简洁优先 | 过度抽象 → 代码难懂 | reviewer 看到 2000 行 diff 想哭 |
| 4. 目标驱动 | 摸鱼摆烂 → 30% 就停 | AI 说"任务完成"实际没完 |

这 4 条不是凭空发明的，是 Andrey 团队被 AI 坑了无数次后总结出来的。

普通程序员用 AI：踩坑 → 抱怨 → 继续踩
顶级程序员用 AI：踩坑 → 总结 → 沉淀成规则，让 AI 别再犯

## 五、怎么用？3 步上手 🚀

1. 在你项目的根目录下创建 `CLAUDE.md` 文件
2. 把上面的 70 行配置原样复制粘贴进去
3. 重启 Claude Code，配置立即生效 ✅

**进阶：按项目定制**

```
# Project-Specific Config (项目定制)
# 保留 4 大原则 [... 上面 70 行 ...]

## 额外规则（按项目加）
# 技术栈
- 本项目使用 TypeScript + React + Next.js
- UI 组件用 shadcn/ui
- 状态管理用 Zustand

# 编码风格
- 不要用 any，必须完整类型
- 函数不要超过 50 行
- 组件不要超过 200 行
- 导出用 named export，不用 default

# 测试
- 必须写单元测试
- 覆盖率不能低于 80%
- 关键路径必须有 E2E 测试

# Git
- commit 信息遵循 Conventional Commits
- 分支命名遵循 feature/xxx
```

通用版 70 行 + 定制版几十行 = 总共 100-150 行，性价比拉满，维护成本几乎为零，但效果能甩别人 10 条街。
