## 📋 面试题目全览

### 一、项目细节与RAG方案 (1~6)

**1. 请详细介绍你参与的那个AI项目的具体细节？**

> 🎯 考察项目深度与真实参与度，需清晰描述项目背景、技术选型、个人职责。

**2. 针对大文档切分，你们是怎么解决语义丢失问题的？**

> 💡 关键是切分策略——固定长度 vs 语义切分，是否使用滑动窗口、段落感知分块等。

**3. 你提到的父子块关联方案，目前进展如何？**

> 🔗 父块（大段落/章节）与子块（小段落/句子）的索引关联，用于兼顾上下文与精确检索。

**4. 现在的RAG方案是不是只支持Markdown格式？图片是怎么处理的？**

> 📄 考察多模态RAG——是否支持PDF/Word/图片，图文的解析与向量化方式。

**5. 你们的图片原始文档是从哪里来的？（关注数据源合规性）**

> ⚖️ 数据合规敏感题——爬取？授权？内部资料？需说明数据溯源与版权处理。

**6. 检索结果的精排依据是什么？**

> 📊 粗排（向量相似度）→ 精排（交叉重排序/RRF/基于规则）的流程。

---

### 二、智能体与记忆管理 (7~11)

**7. 智能体的记忆是如何设计的？**

> 🧠 短期记忆（上下文窗口） vs 长期记忆（向量存储/知识图谱），记忆的压缩与摘要策略。

**8. 如果是多人对话场景，上下文管理逻辑是怎样的？**

> 👥 多会话隔离、用户角色区分、共享知识与私有记忆的分层设计。

**9. 第三个子智能体具体是怎么对接问题处理人的？**

> 🤖 多Agent协作——任务分发、结果聚合、人工兜底（Human-in-the-loop）机制。

**10. 如果模型传参失败，你们有什么具体的处理方案吗？**

> 🔁 异常处理链路：参数校验→重试机制→降级策略→告警通知。

**11. 当前的规划模型在处理垂直领域（如最新期刊专家级内容）时，怎么解决专业知识不足的问题？**

> 📚 领域知识注入：RAG + 专业数据库知识库 + 领域微调 + Few-shot提示。

---

### 三、Skills / Function Calling / MCP (12~14)

**12. Skills具体是怎么工作的？**

> 🛠️ 可复用工具模块定义——输入Schema、执行逻辑、输出格式化，与LLM的调用约定。

**13. Skills 和 Function Calling 有什么区别？**

| 维度 | Skills | Function Calling |
|:---|:---|:---|
| 粒度 | 更高层，含业务逻辑组合 | 底层，单次函数调用 |
| 状态管理 | 可有内部状态/记忆 | 无状态，每次独立调用 |
| 组合性 | 可嵌套组合 | 独立调用 |
| 典型框架 | LangChain Tools | OpenAI Function Call |

**14. MCP 和我们刚才讨论的这些技术架构有什么区别？**

> 🔌 MCP（Model Context Protocol）是标准化的模型-工具通信协议，强调**协议层**的通用性与互操作性，而Skills/FC是**框架层/实现层**的机制。

---

### 四、RAG评估与模型选型 (15~19)

**15. 你们的RAG系统是怎么做效果评估的？**

> 📈 评估框架：端到端（问答准确率） + 模块化（检索召回率、生成忠实度）。

**16. 具体的评估指标有哪些？准确率是多少？**

> 📊 常用指标：Recall@K, MRR, NDCG, Faithfulness, Answer Relevance  
> 💬 *（可分享经验数据，如Top-5召回率85%+等）*

**17. 你们的测试数据里会不会涉及敏感信息？有没有风险？**

> 🔒 数据脱敏：PII过滤、数据分级、最小化原则，合规审计。

**18. 这个分析平台的技术难点在哪里？**

> ⚡ 高并发低延迟检索 + 多模态数据融合 + 动态知识更新 + 评估闭环。

**19. 为什么最终选择了OpenAI的模型？模型选型是怎么考虑的？**

> 🎯 考量维度：任务适配性（推理能力强）、生态成熟度（Function Calling/Tools）、成本-效果权衡、合规性。
