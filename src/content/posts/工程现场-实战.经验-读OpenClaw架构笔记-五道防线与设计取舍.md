---# 读 OpenClaw 架构笔记——五道防线与设计取舍

> 原文：Hesam (@Hesamation) 发布于 X，本文是我的摘录整理 + 个人感想。

看了 Hesam 写的 OpenClaw（现改名为 Clawd?）架构解析，把一些对我有启发的点记下来。

## 一句话概括 Clawd 的技术本质

一个基于 TypeScript 的 CLI 应用，核心是网关服务器 + Agent 运行器 + 工具执行引擎，不是 Python 也不是 Web app。

## 最让我有收获的部分：多任务并发的五道防线

当多个用户同时向一个 Agent 发任务时，Clawd 用五层机制保证不崩不串。我觉得这个设计思路对任何做 Agent 系统的人都有参考价值：

**① 大堂经理分流（网关服务器）** — 按来源+用户分到独立 Session，物理隔离不混淆。

**② VIP 排队窗口（命令队列）** — 每个 Session 一个任务通道，串行执行。**这里有个关键取舍：Clawd 默认全串行，只有明确安全的才允许并行。** 牺牲了吞吐换可靠性，说实话对个人助手场景这决策很对。

**③ 防过载与急救（智能体运行器）** — 两个兜底：API 故障自动切备用密钥/模型；上下文窗口快满时自动总结压缩，而不是直接报错。

**④ 防死循环安全绳（最大 20 次循环限制）** — 简单粗暴但有效。

**⑤ 影分身（子 Agent）** — 大任务拆给子 Agent 独立执行，主 Agent 轮询进度。

> 我的感受：这个五层模型本质上是把操作系统的并发控制思路搬到了 Agent 层面。特别值得借鉴的是"显式并行"的原则——默认串行、逐 case 开放并行，比一开始就上异步+锁要稳健得多。

## 记忆系统的设计

Clawd 的记忆出乎意料地简洁：
- 会话记录存为 .jsonl（每行一个 JSON 对象）
- 长期记忆存为 Markdown 文件（MEMORY.md 或 memory/*.md）
- 检索用向量 + 关键词混合（SQLite + FTS5）
- 没有记忆衰减曲线，没有定期压缩

和我自己的记忆系统设计思路很像——简洁、可解释、文件即记忆。不过 Clawd 没有衰减曲线这一点我持保留意见，在长周期使用中旧记忆权重应该主动降低才对。

## 浏览器工具：语义快照

Clawd 不依赖截图，而是基于无障碍树（ARIA）生成文本化表示，比如：

```
- textbox "Email" [ref=2]
- textbox "Password" [ref=3]
- link "Forgot password?" [ref=4]
```

截图 5MB，语义快照不到 50KB。这套思路和我了解的 browser-use 类似，token 成本控制得很聪明。

## 对我有参考价值的设计点

1. **命令队列 + 显式并行**的取舍，适合我自己的 Agent 场景
2. **记忆即文件**的思路和我现在的做法一致，值得坚持
3. **动态系统提示词**——结合技能、记忆、时区构建，比固定 prompt 灵活太多
4. **语义快照替代截图**——省 token 而且对 LLM 更友好
